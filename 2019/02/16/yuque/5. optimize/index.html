5. optimize

发表于 2019-02-16 |

优化就 2 点：增加打包/编译速度，增加线上访问速度。

写在前面:

  • 所有的工具、手段，都是在一定的性能瓶颈下才引入的，比如并行、缓存等都针对的是比较大的数据，并行工具在处理完后并行任务后需要合并，这是需要消耗时间的，所以记得不要过度优化，只有在有必要时再优化。
  • 优化最需要优化的地方。
  • 不是所有的工具都适合自己的项目。

一个度量打包速度的插件：

  • speed-measure-webpack-plugin

提高编译/打包速度

减少开发过程中的非必要步骤

  • 本地开发中一般无需 babel 转译
  • 本地开发中无需抽离代码，无需 sourceMap
  • 一些代码检查的东西最好放在业务功能完成后统一处理，而不是每次代码修改保存时

缓存

注意哦，在生产环境编译/打包缓存是没有意义的（除非缓存到固定的位置）。

  • cache-loader

缓存 loader 的结果，放在要缓存的其它 loader 之前，如 use: ['cache-loader', 'babel-loader']。

  • hard-source-webpack-plugin

缓存，提升编译速度。

  • thread-loader：优化 loader
  • happypack：优化 loader。
  • parallel-webpack: 适用于多入口

scope hoist 和 tree shaking

两者都是在 ES6 Modules 语法下才能生效的优化（意味着一些转译工具，比如 Babel 需要关闭对 ES6 Modules 的转译），前者是为了提升 JavaScript 的执行速度，后者是减少代码体积（不打包无用代码），所以，为了用到这 2 点优化，开发中应使用 ES6 Module 定义。

webpack4 已经内置了生产模式下的 scope hoist 和 tree shaking 的处理（tree shaking 还要额外在 package.json 中添加 "sideEffects" 配置），如果要在其它场景下开启，可以使用引入 ModuleConcatenationPlugin，另外有一个 tree shaking 支持 commonJs 的工具 webpack-common-shake。

分包和长缓存

打包大小分析工具： webpack-bundle-analyzer

首先一个入口（entry）会对应一个 js 输出，一些第三方库一般都不会修改，应该将这些从业务代码包中抽离，长期缓存，另外一些公共的代码（比如被引用 2 次以上）修改也会比较少，可以单独抽离。webpack 打包出的输出一般有一个统一的出口，默认会放在 vendor 里（如果没有开启分离的话），它一般变动较大，内容比较小，所以独立抽取并 inline （借助 inline-manifest-webpack-plugin 插件）到 html，减少它自身变化对 vendor 等文件的 hash 影响，以及减少请求。

尽量保证每次修改代码不变更不相关文件名里的 hash 值，比如对 css 文件应该使用 contenthash，而不是 chunkhash，后者在 CSS 文件没有修改但如果关联的JS有修改时，hash 值也会改变。
